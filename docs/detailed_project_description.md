# RobotJudge-CI: Detailed Project Description & Architecture

## 1. Introduction and Overview
RobotJudge-CI is an automated, continuous integration evaluation platform designed to systematically validate robotic pathfinding and logic algorithms. The system functions as a strictly deterministic "Online Judge" tailored for robotics software, evaluating untrusted solver submissions against parameterized spatial environments. To ensure absolute parity with rigorous academic and evaluation standards, the architecture has been explicitly engineered to fulfill seven core operational requirements (REQ-1 through REQ-7), bridging the gap between theoretical algorithm design and empirical, statistical validation.

## 2. Execution Engine & Pipeline (REQ-1 & REQ-6)

**Requirement Statement:** The system must utilize container-first execution to provide secure, ephemeral containment isolation, and enforce strict asynchronous resource and time constraints on all evaluations.

**Implementation Details:**
To guarantee that untrusted student or researcher code cannot compromise the host machine or web server, the RobotJudge-CI pipeline strictly forbids native host execution. We developed a distinct `rj-sandbox` Docker environment (via `Dockerfile.sandbox`) running a highly restricted, unprivileged non-root user account. 

When a submission ingress occurs, the `robotjudge/runner.py` module orchestrates the evaluation via a dynamic `--sandbox` flag. This command instantly spins up a secure, network-isolated container, logically mapping only the necessary problem grid inputs via read-only volume mounts.

Crucially, this containerization enforces strict hardware-level execution constraints (REQ-6). The runner dynamically limits CPU allocation and tracks peak memory utilization utilizing the Python `psutil` library. Furthermore, explicit wall-clock Time Limit Exceeded (TLE) parameters are enforced asynchronously. If a solver algorithm enters an infinite loop or breaches its latency limits, the Execution Engine aggressively terminates the container, preventing hardware monopolization and ensuring the continuous integration queue remains unblocked. To compensate for Docker daemon spin-up overhead, the Engine mathematically subtracts initialization latency to ensure fair millisecond-accurate runtime recording.

## 3. Seed Governance & Statistics (REQ-2 & REQ-4)

**Requirement Statement:** The architecture must abstract seed management from the submission script, dictate exact sequence boundaries, and compute holistic statistical verdicts—such as 95th Percentile computations—rather than anecdotal single-run successes.

**Implementation Details:**
RobotJudge-CI natively supports highly parameterized evaluation workflows. The algorithmic agent submitted by the user is entirely blind to the configuration logic. Instead, the web Submission Interface (`web/static/submit.html`) governs the execution, providing users with absolute control over "Seed Range" inputs (e.g., executing seeds `0` through `4` sequentially).

The CI backend iteratively runs the containerized solver against each individual seed variant. Upon completion, the aggregation mathematical engine within `robotjudge/grader.py` compiles the distributed data points. It does not merely report pass/fail; it programmatically derives aggregate Suite Success Rates across the entire topological variance. 

For rigorous performance analysis, the system calculates and exposes explicit percentile mathematics. It mathematically calculates the **P95 (95th Percentile) Runtime** and the **Mean Execution Cost** across all simulated seeds. These aggregated metrics, alongside granular per-seed status breakdowns, are projected directly to the Dashboard UI, providing users with a deep statistical verdict on their algorithm's stability and variance.

## 4. Oracle / Validator Separation (REQ-3)

**Requirement Statement:** Environment formulation logic must operate distinctly from the path verification mechanics, avoiding reliance on the generator pipeline for trajectory validation.

**Implementation Details:**
A foundational design principle of RobotJudge-CI is the absolute decoupling of the Scenario Generator from the Validation Oracle. 

The `robotjudge/generator.py` module acts strictly as a deterministic environment provisioner. It interprets static JSON schema matrices and dynamic pseudo-random seeds to layout grid obstacles, generating the exact `case.json` topological map.

Post-execution, the evaluation is handled by an entirely independent Oracle composed of `robotjudge/schema.py` and `grader.py`. The Oracle acts mathematically. It ingests the raw coordinate trajectory logs (`path.json`) generated by the solver and overlays them against the boundaries of the environment matrix. It validates sequence costs, geometric path length, and movement legality strictly through mathematical heuristic penalties, completely ignorant of the logic that generated the maze initially. This ensures objective verification.

## 5. Artifact & Traceability (REQ-5)

**Requirement Statement:** The system must retain permanent, schema-level tracking of all submission attempts, storing high-fidelity metric derivations for full traceability.

**Implementation Details:**
Transient standard-output logs are insufficient for reproducible academic evaluations. RobotJudge-CI utilizes a robust relational persistence layer via PostgreSQL (`web/database.py`). 

Every invocation is permanently recorded within the Result Store schemas (`Run`, `ResultSet`, `PathRecord`). The database captures perfect historical snapshots including exact exit codes, error streams, memory footprints, and JSON coordinate artifacts. 

Furthermore, to guarantee dataset provenance during longitudinal studies or iterative algorithm improvements, the Problem Registry implements strict dataset versioning. Every problem instance generated and stored in the database is stamped with an explicit schema version (e.g., `v1.0`), ensuring that performance regressions can be accurately traced back to either algorithm decay or explicit environmental map updates.

## 6. Safety Constraints (REQ-7)

**Requirement Statement:** The pipeline must enforce proactive physical boundary checks including grid collision states and kinematic safety bounds to guarantee executable trajectories.

**Implementation Details:**
While the current prototype does not require a full 3D physics-engine implementation (such as Gazebo or Webots), it strictly forbids impossible physical trajectories. 

The validation Oracle goes beyond simple geometric collision detection (i.e., verifying the robot did not path through an impassable wall). It mathematically implements explicit **Kinematic Safety Constraints** directly within the validation loop. 

Submissions are analyzed iteratively to ensure no path vectors attempt impossible real-world mechanics. Specifically, the schema validator flags any instantaneous 180-degree grid inversions (e.g., moving North and immediately pivoting South within the same heuristic step). Because such a maneuver would require an infinite angular acceleration threshold in a physical robot chassis, the system immediately fails the run as a "Kinematic Safety Violation." This guarantees that all successful paths derived within the grid ontology can be smoothly interpolated by physical robotic hardware in future ROS2 deployments.
