System Operations Manual

Comprehensive Guide to the RobotJudge-CI Continuous Integration & Automated Evaluation Framework

I. Algorithmic Dataset Exploration

1.1 Problem Dashboard Architecture

The primary interface serves as a comprehensive registry of pathfinding challenges. Key functionalities include:

  • Multi-Tier Filtering & Version Control (REQ-5): Segregate testcases by computational complexity (Easy, Medium, Hard) to target specific algorithmic stress points. Every topology is strictly version-controlled (e.g. v1.0) to guarantee longitudinal reproducibility across evaluation cycles.
  • Dynamic Metadata Sorting: Sort cases by identifier, obstacle family, grid dimensions, or search budget to identify outliers.
  • Universal Search: Instant lookups across Case IDs and environment categories (e.g., "maze", "trap").

1.2 Visual Diagnostic Environment

Selecting a specific Case ID initiates the high-fidelity visualization layer:

  • Spatial Rendering: Interactive grid display highlighting start (S) and goal (G) coordinates.
  • Kinematic Safety Analysis (REQ-7): Detailed view of movement constraints. The validation Oracle explicitly enforces mathematical safety bounds, ensuring output trajectories do not contain impossible physical maneuvers (e.g., infinite acceleration 180-degree reversals).

II. Solver Evaluation and Integration

2.1 Submission Modalities

RobotJudge-CI supports a dual-path submission system designed for both baseline comparison and proprietary development:

  • Intrinsic Baseline Selection: Immediate execution of pre-baked reference algorithms (BFS, Dijkstra, A*) to establish performance benchmarks.
  • External Script Injection: Uploading of arbitrary .py files. The system dynamically injects these into the evaluation container.

2.2 Real-Time Execution Lifecycle

Upon initialization via the "Run Solver" trigger, the system manages a complex asynchronous execution flow:

  • SSE-Driven Progress Synchronization: The backend utilizes Server-Sent Events to provide sub-second updates on evaluation progress.
  • Sandboxed Execution: Solvers are executed in an isolated environment with enforced CPU time limits.
  • Live Result Injection: As testcases are completed, metrics are programmatically committed to the PostgreSQL core and rendered in the progress view.

III. Performance Analytics and Reporting

3.1 Longitudinal Metric Analysis

The "Status" module provides a high-level overview of historical solver efficacy:

  • Statistical Success Aggregation (REQ-4): Percentage-based metrics detailing testcase coverage within time/cost budgets across all stochastic seeds.
  • Computational Variance (P95): Programmatic calculation of the 95th Percentile latency (P95 Runtime) to provide deep statistical insights into worst-case temporal stability.
  • Path Optimality: Comparative cost metrics ensuring the algorithm identifies the globally optimal trajectory without safety infractions.

IV. Continuous Integration (CI) and Automated Validation

4.1 Core CI Philosophy: The Automated Gatekeeper

RobotJudge-CI is fundamentally architected as a Continuous Integration (CI) platform for robotics pathfinding. It shifts algorithmic evaluation from manual, ad-hoc local execution to a fully automated pipeline driven by code commits and pull requests (e.g., via GitHub Actions or GitLab CI).

  • Automated Regression Testing: Every code push automatically spawns the isolated headless execution engine, running the solver against the entire Versioned Problem Registry to proactively catch algorithmic bugs.
  • Performance Benchmarking Gate: The CI pipeline acts as a strict programmatic gatekeeper. If a commit violates the absolute runtime constraints (REQ-6) or causes a regression in average path cost, the CI job automatically fails the build.
  • Traceable Artifact Generation: The native CI workflow records structured P95 metrics, deterministic logs, and visual path artifacts per-seed, establishing a longitudinal performance history that can be audited over time (REQ-5).
